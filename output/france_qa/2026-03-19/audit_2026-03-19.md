### 1.2 CRITICAL: SAI Race 2 (900m Turf) — absurd figures (179–214)

Winner RAFAGA: raw figure **294.36**, calibrated **214.11**. The finishing time of 53.71s for 900m on "Très souple" is ~19s below the standard of 73.14s. The going allowance is **negative** (-0.0101 s/m) — physically impossible on soft ground.

**Root cause:** The real-time GA computation (live_ratings.py `_compute_realtime_ga`) was distorted by this race's own anomalous data feeding back into the meeting-level GA. The race should have been caught by QC-2 (impossible pace check) but 53.71s/900m = 0.0597 s/m = 12.0 s/f, which falls within the valid 10–18 s/f range. The time itself is plausible for a sprint on fast ground — the problem is the **standard time of 73.14s is wrong** (that pace = 14.7 s/f, far too slow for 900m).

**Fix needed:** Add QC check: suppress any race where `|raw_figure| > 150` or where `corrected_time / standard_time < 0.85`.

### 1.3 AVE R1 & GPD R7/R8/R9 — skipped due to QC-3 (all beaten lengths = 0)

**Root cause identified:** All 4 skipped races have `distanceCumulative = 0.0` for every non-winner. The pipeline's QC-3 check (`all beaten lengths identical`) correctly triggers and suppresses these races. The beaten-length data from PMU was either missing or all zeros for these races.

This is NOT a bug — the QC check is working as designed. The issue is upstream: PMU didn't provide beaten-length data for these races. AVE R3 passes QC-3 because it has real beaten lengths (0.75L, 2.0L, 3.25L, etc.).

### 1.4 AVE R3 — deeply negative figures (-9 to -25)

If the going was "Très souple" (as at SAI), the GA correction of ~0.52 s/f would account for most of the deviation.

## Part 2 — Code Bugs

### 2.1 CRITICAL: GA fallback value uses wrong unit (s/f instead of s/m)

**Files:** `speed_figures.py:826`, `live_ratings.py:95`

```python
# speed_figures.py line 826
prior_ga = FRANCE_GOING_GA_PRIOR.get(going_desc, 0.05)  # BUG: 0.05 s/f, not s/m

# live_ratings.py line 95
return FRANCE_GOING_GA_PRIOR.get(going_desc, 0.05)  # BUG: same
```

All `FRANCE_GOING_GA_PRIOR` values are in **s/m** (divided by 201.168). The fallback `0.05` is in **s/f** — approximately **100x too large**. Any going description not in the dictionary would get a GA prior of 0.05 s/m = 10.06 s/f, causing:
- Bayesian shrinkage to massively inflate GAs for meetings with unmapped going descriptions
- `estimate_going_allowance()` to return absurd values as a last fallback

**Impact:** Latent bug — triggers for any novel going-description casing/spelling not in the dictionary. All known going descriptions are mapped, but data from new sources could trigger it.

**Fix:** Change both to `0.05 / 201.168` or `FRANCE_GOING_GA_PRIOR.get("Bon", 0.10 / 201.168)`.

### 2.2 CRITICAL: Batch pipeline clips raw figures but live pipeline does not

**speed_figures.py:1000-1003:** Raw winner figures are clipped to [-50, 250].
**live_ratings.py:** No clipping applied. SAI R2 produced a raw figure of 294 (which would be clipped to 250 in batch). This allowed the absurd 214 calibrated figure through.

**Fix:** Add `winners["raw_figure"] = winners["raw_figure"].clip(lower=-50, upper=250)` after line 342 in live_ratings.py.

### 2.3 HIGH: GA race-level attenuation only in live pipeline

**live_ratings.py:323-338** implements per-race GA attenuation when meeting GA overshoots individual race deviations. **speed_figures.py** does not have this. This creates systematic divergence between batch and live figures for the same historical data.

**Recommendation:** Either add the attenuation to the batch pipeline or remove it from live — they should be consistent.

### 2.4 HIGH: Velocity-weighted LPL has no bounds on velocity ratio

**speed_figures.py:1053** and **live_ratings.py:231**: `velocity_ratio = (standard_time / winner_time).clip(0.85, 1.15)`. The clip range of [0.85, 1.15] allows a 15% velocity adjustment to LPL. For a race on extreme going where the winner runs 20% slower than standard, the ratio would be clamped at 0.85, but LPL would still be reduced by 15% — artificially compressing the beaten-length penalties. Consider whether this feature adds value or just introduces noise.

### 2.5 HIGH: Non-finisher handling in beaten lengths

**beaten_lengths.py:123**: NaN margins are treated as 0 in the cumulative sum. If a non-finisher has no beaten-length text (NaN), they get `distanceCumulative` equal to the previous horse's cumulative BL — making them appear to have finished just behind. Non-finishers should be excluded entirely or given NaN cumulative BL.

**database.py**: No `runner_status` column exists to distinguish finishers from non-finishers (fell, pulled up, refused, etc.).

### 2.6 MEDIUM: `_FRACTION_MAP` silently maps unknown fractions to 0

**beaten_lengths.py:72**: `_FRACTION_MAP.get(m.group(2), 0.0)`. If PMU provides "3/8 L" or "1/5 L", the fraction evaluates to 0.0 silently — data corruption rather than a flagged parse failure.

### 2.7 MEDIUM: Dead heat handling incorrect

**beaten_lengths.py:112-126**: When two horses share the same `positionOfficial`, `sort_values` gives arbitrary order. The second dead-heater's margin gets added on top of the first's cumulative BL, producing incorrect totals.

### 2.8 HIGH: WFA function returns `None` for invalid ages

**constants.py `get_france_wfa_allowance`**: For `age < 2` or `NaN` age, none of the if-branches match and the function implicitly returns `None`. When added to a float figure in `apply_wfa_adjustment`, this causes a `TypeError`. Needs an explicit `return 0.0` fallback at function end.

### 2.9 MEDIUM: Stale "Bon" GA fallback in standard time prior correction

**speed_figures.py:310**: `FRANCE_GOING_GA_PRIOR.get("Bon", 0.04 / 201.168)` — the hardcoded fallback `0.04` is a stale value from before the empirical update. The actual constant in `constants.py` is `0.10 / _M`. Affects the prior correction for any winner whose going description is missing from the dictionary when computing standard times.

### 2.10 MEDIUM: 2yo-only detection operates on winners-only subset

**speed_figures.py:94-102**: `_filter_std_time_winners` checks for 2yo-only races by grouping on `race_id` within the winners dataframe. Since there's one winner per race, `race_max_age == race_min_age == 2` is just checking whether the winner is age 2 — not whether the race is restricted to 2yos. A mixed-age race won by a 2yo would be incorrectly excluded from standard time computation.

### 2.11 LOW: Dead code and unused imports

- **speed_figures.py:1282**: `_compute_bl_band_corrections()` is defined but never called (commented out at line 1230).
- **speed_figures.py:40**: `FRANCE_GOOD_GOING` is imported but never referenced.
- **live_ratings.py**: `ga_se_dict` loaded from artifacts but never used; email functionality defined but not wired to CLI.
- **constants.py**: `GA_CONVERGENCE_TOL` and `INTERPOLATED_GA_WEIGHT` defined but never imported or used.

## Part 3 — Architecture Issues

### 3.1 HIGH: Live vs batch calibration divergence

The live pipeline uses **batch-derived** calibration params (scale/shift), but the live pipeline has additional features (GA race attenuation) and missing features (no raw figure clipping). When the batch is re-run with different data, the calibration params change, but the live pipeline's structural differences mean figures won't match even with identical params.

**Recommendation:** Extract the figure computation into a shared function used by both batch and live pipelines.

### 3.2 HIGH: `figure_comment` masks distinct failure modes

All quality-check failures produce the same comment: "no historical data to generate figures". This loses diagnostic information — a missing standard time (QC-1) is a very different issue from broken beaten lengths (QC-3) or impossible pace (QC-2).

**Recommendation:** Use distinct comments: "missing standard time", "impossible finishing time", "broken beaten lengths".

### 3.3 MEDIUM: Individual runner times discarded

**field_mapping.py:80-182**: `RunnerRow.time_seconds` is queried from the database but never mapped to any output column. Only winner times are used. If PMU provides individual times for some runners, this data is lost.

### 3.4 MEDIUM: No surface column stored in RaceRow

**database.py**: Surface is inferred from `going` text and `parcours` at query time. If the inference logic changes, historical results would be re-classified. Storing surface explicitly would be more robust.

## Part 4 — Data/Artifact Issues

### 4.1 Standard times — thin samples

Current `standard_times.csv` has 722 rows (down from 800 referenced in older audits — non-French contamination has been cleaned):
- 27.1% of rows (196) have n < 20
- 16.9% (122 rows) have n < 15
- 36 rows at the minimum n=10
- 53 rows flagged `provisional=True` (high median/mean divergence)
- SAI_900_Turf (the source of the absurd SAI R2 figures) needs its sample size verified

### 4.2 Non-French course contamination — RESOLVED

The `NON_FRENCH_COURSE_CODES` blocklist has been effective. All 20 non-French course codes (ASC, DON, EPS, GOO, etc.) are **absent** from the current `standard_times.csv`. The 722 rows contain 113 unique French course codes only.

GPD is NOT in the blocklist and appears to be a valid French course. GPD AW entries have healthy sample sizes (n=26-130) with low divergence. However, **GPD Turf long-distance standards are unreliable**: GPD_1800_Turf (n=14, 3% divergence, provisional), GPD_2200_Turf (n=10, 3.6% divergence, provisional), GPD_2400_Turf (n=15, 5.2% divergence, provisional). GPD_1600_Turf has a suspiciously round median of exactly 93.51s (most rows have 14+ decimal places).

### 4.3 Going allowance calibration bias

From the existing GA audit: mean Turf GA = +0.248 s/f, indicating standards are calibrated ~2s too fast at 8f. This has been partially addressed by the prior-based correction in `compute_standard_times()`, but residual bias remains.

### 4.4 Cross-course outliers

XTD entries are the most problematic in the standard times file — ~10% median/mean divergence at XTD_1400_Turf and ~8% at XTD_1800_Turf, flagged as both provisional and cross-course outliers. Other outliers include D04, D15, F03, LIG, OST, SA7, VPR.

### 4.5 Going allowance file integrity

`going_allowances.csv` has 11,886 rows in seconds-per-metre (`going_allowance_spm` column). 13.3% of meetings have negative GA (faster than standard). 2026-03-19 entries for GPD_AW (0.000573 spm) and SAI_Turf (0.001224 spm) are within plausible range. The last ~200 rows break chronological order (backfills/late additions).

## Part 5 — Constants Review

### 5.1 Going description casing variants

`FRANCE_GOING_GA_PRIOR` has entries for "Bon Léger" (0.02/201.168) and "Bon léger" (0.11/201.168) — these map to **different values** (0.02 vs 0.11 s/f). This is likely intentional (different empirical means observed for different casing in PMU data), but it means casing inconsistency in source data would produce different GA priors for the same going description.

### 5.2 WFA table coverage

The 3yo WFA table has columns for 5f, 6f, 7f, 8f, 10f, 12f. French races at 9f (1800m) interpolate between 8f and 10f. Races at distances beyond 12f (e.g., 2400m ≈ 12f, 3000m ≈ 15f) use the 12f value (0 in most months), which may underestimate the WFA at extreme distances.

### 5.3 BL attenuation threshold

`BL_ATTENUATION_THRESHOLD = 20.0` is static, but the live pipeline (line 377) uses a going-dependent threshold: `T = clip(20 + GA * -8, 10, 30)`. The batch pipeline also does this (line 1066). Both are consistent.
